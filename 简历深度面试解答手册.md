### 1.1 语言基础：JavaScript (ES6+) / TypeScript / HTML5 / CSS3 / Sass / Less

### 1. ES6+ 你常用哪些特性？举几个实际场景。

<details>
<summary>查看答案</summary>

- **解构赋值**：从接口返回的对象中提取字段，如 `const { name, price, images } = product`，避免重复写 `product.xxx`
- **可选链 `?.` 与空值合并 `??`**：处理接口数据缺失，如 `supplier?.company?.name ?? '未知供应商'`
- **Promise.all / Promise.allSettled**：在首页多楼层并行请求时，用 `Promise.allSettled` 保证单个楼层接口失败不影响其他楼层渲染
- **Proxy / Reflect**：在 Vue 3 响应式原理中理解其底层机制；在表单场景中用 Proxy 做数据变更拦截
- **WeakMap / WeakSet**：用来做不阻止垃圾回收的对象缓存，比如 DOM 节点到业务数据的映射、组件内部临时缓存，避免内存泄漏
- **Iterator / for...of / Generator**：在轨迹回放功能中，用 Generator 逐帧产出轨迹点数据，控制播放节奏
- **模板字符串**：拼接动态 URL、GraphQL 变量或错误日志，如 `/api/products/${id}`，避免 `+` 号字符串拼接
- **箭头函数**：在数组 `map/filter/reduce`、Promise 回调中简化写法并保持 `this` 指向，如 `list.filter(item => item.status === 'online')`
- **默认参数 / Rest / Spread**：函数参数给默认值减少判空；`...` 展开合并配置或数组，如 `{ ...defaultConfig, ...userConfig }`、`Math.max(...scores)`
- **Async / Await**：把多层 Promise `.then()` 改成同步写法，提升可读性，如并发获取多个楼层数据 `const floors = await Promise.all(floorIds.map(fetchFloor))`
- **Class 语法**：封装业务实体或工具类，如定义 `FloorModel`、`Device` 等数据模型，配合继承实现通用逻辑复用
- **Map / Set**：`Map` 用于复杂对象键值的缓存映射；`Set` 做数组去重，如 `const uniqueTags = [...new Set(tags)]`

</details>

### 2. TypeScript 在项目里怎么用的？泛型用过吗？

<details>
<summary>查看答案</summary>

- 在 Alibaba.com 全球供给项目中，React + TS 是主技术栈。所有楼层组件的 props、接口返回数据、GraphQL Query 结果都有严格类型定义
- 泛型简单说就是"类型参数"——写代码时不写死具体类型，用的时候再指定，让代码既灵活又安全。
- 泛型的典型场景：封装通用 API 请求函数 `function fetchData<T>(query: string): Promise<ApiResponse<T>>`，T 根据楼层数据类型动态指定
- 用 `Record<string, FloorConfig>` 做楼层配置映射，用 `Partial<T>` 做编辑表单的类型推导
- 在 GraphQL 场景下，通过 codegen 工具根据 Schema 自动生成 TS 类型，保证前后端类型一致

</details>

### 3. Sass 和 Less 的区别是什么？项目里怎么选？

<details>
<summary>查看答案</summary>

- Sass 用 `@mixin` / `@include` 做样式复用，支持函数、条件判断，生态更成熟（Element Plus 用的就是 Sass）
- Less 语法更接近 CSS，用 `@mixin` 较少，主要靠 `mixin` 函数，Ant Design 使用 Less 做主题定制

- 两者核心差异不大，关键是跟着组件库的生态走

</details>

### 1.2 前端框架：精通 Vue 2/3 全家桶（Vue Router / Pinia / Vuex），熟练 React 18 + Hooks 生态（ice.js）

### 4. Vue 2 和 Vue 3 的核心区别？

<details>
<summary>查看答案</summary>

- **响应式原理**：Vue 2 用 `Object.defineProperty` 劫持 getter/setter，Vue 3 用 `Proxy` 代理整个对象，支持 Map/Set/WeakMap 等数据结构，且可以监听属性新增和删除
- **Composition API**：Vue 3 提供 `setup()` + `ref/reactive/computed/watch`，可以把相关逻辑聚合在一起（而不是按 data/methods/computed 分散），利于逻辑复用和代码组织
- **生命周期变化**：使用 Composition API 时，`setup()` 在 `beforeCreate` 之前执行，因此 `beforeCreate/created` 中的逻辑通常可直接写在 `setup()` 的同步代码中；`beforeDestroy/destroyed` 改名为 `beforeUnmount/unmounted`
- **性能优化**：Vue 3 引入静态提升（hoistStatic）、补丁标记（PatchFlags）、Block Tree，diff 算法只比较动态节点，渲染性能提升显著
- **Tree-shaking**：Vue 3 全局 API 改为按需导入（如 `import { createApp } from 'vue'`），打包体积更小

</details>

### 5. Pinia 和 Vuex 的区别？为什么选 Pinia？

<details>
<summary>查看答案</summary>

- Pinia 是 Vue 官方推荐的状态管理，去掉了 Vuex 的 `mutations`，只有 `state/getters/actions`，更简洁
- 天然支持 TypeScript，state 类型自动推导
- 模块化不需要 `modules` 嵌套，每个 store 独立文件，按需引入
- 支持插件扩展（如持久化插件 `pinia-plugin-persistedstate`）
- 在 OMS 项目中从 Vuex 迁移到 Pinia，store 文件数量没变，但代码量减少约 30%，去掉了大量 `commit` 的 mutations 样板代码

</details>

### 6. Vue Router 路由守卫你怎么用的？

<details>
<summary>查看答案</summary>

- **全局前置守卫 `router.beforeEach`**：做登录鉴权，检查 token 是否存在，不存在跳转登录页；检查角色权限，无权限跳转 403
- **路由独享守卫 `beforeEnter`**：特定路由需要额外校验，如设备详情页需要先验证设备 ID 合法性
- **组件内守卫 `onBeforeRouteLeave`**（Vue 3）：表单编辑页离开前提示用户"是否保存未保存的更改"
- 在 OMS 项目中，结合 OA 单点登录的 token，在全局守卫中做 token 有效性校验 + 自动刷新

</details>

### 7. React 18 的 Hooks 你用哪些？和 Vue 3 Composition API 有什么区别？

<details>
<summary>查看答案</summary>

- 常用：`useState`、`useEffect`、`useMemo`、`useCallback`、`useRef`、`useContext`、自定义 Hooks
- 在 Alibaba.com 项目中，ice.js 框架提供 `useData`、`useSearchParams` 等内置 Hooks，结合 React 18 的 `useTransition` 做非阻塞渲染（比如切换楼层 Tab 时标记为低优先级更新）
- **核心区别**：
  - Vue 3 的 `reactive` 直接修改属性即可触发更新；`ref` 需要修改 `.value`（如 `count.value++`）。React 的 `useState` 需要调用 `setState` 触发更新
  - Vue 3 的 `computed` 自动追踪依赖，React 的 `useMemo` 需要手动声明依赖数组
  - Vue 3 的 `watch` 可以精确监听某个属性变化，React 的 `useEffect` 是依赖整体对比
  - React Hooks 容易出现闭包陷阱（stale closure），常用 `useRef`、函数式更新或正确填写依赖来解决；Vue 3 的响应式值由框架代理，组件重新执行时读取的是最新值，闭包问题相对少很多，但在 Composables 中也要注意不要提前解构出普通值

</details>

### 1.4 工程化：Vite / Webpack / pnpm / ESLint / 按需导入 / 路由懒加载 / 分包策略

### 10. Vite 比 Webpack 快在哪里？

<details>
<summary>查看答案</summary>

- **开发环境**：Vite 开发环境基于原生 ESM，不做打包，浏览器直接按需请求模块文件，启动速度通常是毫秒级（Webpack 开发环境需要先打包整个依赖图）
- **热更新 HMR**：Vite 只更新变化的模块（利用 ESM 的 `import.meta.hot`），Webpack 需要重新编译整个模块依赖图
- **预构建**：Vite 用 esbuild 做依赖预构建（将 CommonJS/UMD 转为 ESM），速度比 Webpack 快 10-100 倍
- **生产构建**：Vite 用 Rollup 打包，输出 Tree-shaking 效果更好的 ESM 格式

</details>

### 11. 按需导入你是怎么做的？打包体积减少了多少？

<details>
<summary>查看答案</summary>

- Element Plus 按需导入：用 `unplugin-vue-components` + `unplugin-auto-import` 插件，自动扫描模板中使用的组件并按需导入对应样式
- 原理：插件在编译时解析 `<el-button>` 等标签，自动注入 `import { ElButton } from 'element-plus'` 和对应 CSS
- 不需要手动写 `import` 语句，也不需要在 main.js 中全量引入
- 在 OMS 项目中，从全量引入 Element Plus（约 700KB CSS）改为按需导入后，CSS 体积大幅减少（约 60%-80%，具体取决于实际使用的组件数量）

</details>

### 12. 路由懒加载怎么实现的？

<details>
<summary>查看答案</summary>

```js
// 不使用懒加载（同步导入，所有页面打包在一个 chunk 里）
import DeviceManage from "@/views/device/index.vue";

// 使用懒加载（动态 import，访问时才加载）
const DeviceManage = () =>
  import(/* webpackChunkName: "device" */ "@/views/device/index.vue");
```

- 在 Vite 中直接使用 `() => import('./views/xxx.vue')`，Vite 会自动做代码分割
- 原理：`import()` 是 ES 的动态导入语法，返回 Promise，Webpack/Vite 会将该模块单独打成一个 chunk
- 结合 `webpackChunkName` 魔法注释（Webpack）或 Rollup 的 `manualChunks`（Vite），可以控制分包粒度

</details>

### 13. 分包策略具体怎么配的？说说你的思路。

<details>
<summary>查看答案</summary>

在 Vite 中通过 `build.rollupOptions.output.manualChunks` 配置：

```js
// vite.config.js
export default {
  build: {
    rollupOptions: {
      output: {
        manualChunks: {
          "vendor-vue": ["vue", "vue-router", "pinia"], // Vue 核心库
          "vendor-ui": ["element-plus"], // UI 组件库
          "vendor-utils": ["axios", "dayjs", "lodash-es"], // 工具库
          "vendor-map": ["ol", "proj4"], // 地图库（体积大，单独分包）
        },
      },
    },
  },
};
```

**分包思路：**

1. **核心框架单独分包**（vue/vue-router/pinia）：这些库几乎不变，利用浏览器缓存，用户首次加载后后续访问直接用缓存
2. **UI 组件库单独分包**（element-plus）：体积大且更新频率低于业务代码
3. **重型库单独分包**（OpenLayers、ECharts）：只在特定页面使用，避免加载到首屏
4. **业务代码按路由自动分割**：结合路由懒加载，每个页面自动一个 chunk
5. **效果**：首屏只需加载 `vendor-vue` + `vendor-ui` + 当前页面 chunk，从原本未分包的较大 bundle 降至首屏仅加载约 2M 的核心资源（具体数值与项目总代码量和依赖规模有关）

</details>

### 14. pnpm 和 npm/yarn 的区别？为什么选 pnpm？

<details>
<summary>查看答案</summary>

- **硬链接 + 符号链接**：pnpm 将依赖安装到全局 store（`~/.pnpm-store`），项目 node_modules 里只有符号链接，节省磁盘空间
- **严格的依赖隔离**：pnpm 不允许访问未在 `package.json` 中声明的依赖（幽灵依赖），避免"我这里没装但能用"的隐患
- **安装速度**：得益于硬链接机制，安装速度比 npm/yarn 快 2-3 倍
- **Monorepo 支持**：通过 `pnpm workspace` 管理多包项目，比 Lerna 更轻量

</details>

### 1.5 可视化：OpenLayers / Leaflet / 高德地图 / WebGL / Canvas / ECharts / Recharts

### 15. OpenLayers 和 Leaflet 的区别？为什么不同项目用不同的？

<details>
<summary>查看答案</summary>

- **OpenLayers**：功能全面，支持多坐标系（通过 proj4 转换）、多图层叠加、高级渲染（WebGL 图层）、复杂交互（绘制、测量），适合专业 GIS 场景
- **Leaflet**：轻量（约 40KB），API 简洁，适合简单的地图展示和标记，插件生态丰富
- OMS 项目用 OpenLayers 因为需要：多坐标系转换（WGS84 → GCJ02）、10W+ 海量点位渲染、轨迹回放动画、多图层管理
- 全球机会洞察平台用 Leaflet 因为只需要全球营销活动分布的简单地图视图，轻量即可

</details>

### 16. WebGLPointsLayer 怎么做到 10W+ 点位渲染的？FPS 如何保证？

<details>
<summary>查看答案</summary>

- OpenLayers 默认的 Canvas 渲染器在点位超过 1W 时就会明显卡顿，因为每个 Feature 都会触发独立的 Canvas 绘制调用，CPU 开销随点位数线性增长
- `WebGLPointsLayer` 使用 WebGL 直接在 GPU 上批量渲染所有点位，一次 drawCall 绘制所有点
- **关键优化手段：**
  1. **视口裁剪**：只渲染当前视口内的点位，地图移动时动态更新可视区域
  2. **LOD（Level of Detail）**：缩放级别低时聚合显示（Cluster），缩放级别高时展开单个点
  3. **属性缓冲区**：点位数据（坐标、颜色、大小）预计算存入 WebGL Buffer，避免每帧重复计算
  4. **Web Worker 预处理**：坐标转换（proj4）放在 Worker 线程中，不阻塞主线程
- **FPS 保障**：10W+ 点位在 WebGL 模式下通常可保持 15-30 FPS（取决于硬件、点位样式和渲染策略），而 Canvas 2D 在同等数据量下通常只有 1-5 FPS，差距非常明显

</details>

### 17. 轨迹回放怎么做的？速度/方向动画怎么实现？

<details>
<summary>查看答案</summary>

```
核心思路：按时间轴插值轨迹点 → 更新标记位置 → 计算朝向角 → 旋转图标
```

- **数据准备**：后端返回带时间戳的轨迹点数组 `[{lon, lat, speed, timestamp, heading}]`
- **插值算法**：根据播放倍速计算当前时间点，用线性插值在两个相邻轨迹点之间计算当前位置
- **方向动画**：通过 `heading`（航向角）或相邻两点计算方位角 `Math.atan2(dy, dx)`，设置图标旋转角度
- **速度展示**：用颜色渐变表示速度变化（绿色→黄色→红色），或在线段上叠加速度-时间折线图
- **性能**：使用 `requestAnimationFrame` 驱动动画循环，避免 `setInterval` 的精度问题和性能浪费
- **拖拽进度条**：支持拖动进度条跳转到任意时间点，内部维护一个当前帧索引

</details>

### 18. ECharts 和 Recharts 的使用场景分别是什么？

<details>
<summary>查看答案</summary>

- **ECharts**：在 OMS 项目中使用，优势是功能强大，支持地图热力图、复杂联动、大数据量（dataZoom 组件），配置项丰富

### 1.6 音视频：WebSocket / WASM / requestAnimationFrame / WebGL 渲染

### 19. 无插件视频流方案的具体实现原理？

<details>
<summary>查看答案</summary>

这是简历中最重要的技术亮点之一，完整链路如下：

```
摄像头/录像机 → 流媒体服务器 → WebSocket → 前端接收二进制流 → WASM 解码 → 视频帧 → WebGL 渲染到 Canvas
```

**详细步骤：**

1. **WebSocket 连接**：与流媒体服务器建立 WebSocket 长连接，接收 H.264/H.265 编码的二进制视频流
2. **WASM 解码**：将 FFmpeg 的解码核心编译为 WebAssembly（.wasm 文件），在浏览器中加载 WASM 模块，把 H.264 码流解码为 YUV 原始帧数据
3. **YUV → RGB 转换**：在 WASM 内部或 WebGL Shader 中做色彩空间转换（YUV420 → RGBA）
4. **requestAnimationFrame 驱动**：用 rAF 做帧调度，每帧从解码缓冲区取出一帧，确保和浏览器刷新率同步，避免掉帧
5. **WebGL 渲染**：创建 WebGL 纹理（Texture），将 RGB 帧数据上传到 GPU，用 Fragment Shader 渲染到 `<canvas>` 上
6. **为什么用 WebGL 而不是 Canvas 2D**：WebGL 是 GPU 渲染，比 `ctx.putImageData()` 快一个数量级，尤其是 1080P 分辨率下

</details>

### 20. 为什么替代 Windows 插件？原来的方案有什么问题？

<details>
<summary>查看答案</summary>

- 原方案是海康的 ActiveX / NPAPI 浏览器插件，**只能在 Windows + IE/旧版 Chrome** 上运行
- 问题：不支持 Mac/Linux、不支持新版 Chrome（NPAPI 已被废弃）、安装复杂、安全漏洞风险
- 新方案基于 Web 标准（WebSocket + WASM + WebGL），**全平台全浏览器兼容**，用户无需安装任何东西

</details>

### 21. 延迟大概多少？怎么处理延迟？

<details>
<summary>查看答案</summary>

- 端到端延迟约 200-500ms（取决于网络和分辨率）
- **降低延迟的手段：**
  1. 解码缓冲区控制：只缓存最近 2-3 帧，丢弃过期帧，避免累积延迟
  2. 关键帧请求：当延迟过大时，向服务端请求 I 帧（关键帧），丢弃 P/B 帧等待下一个 GOP
  3. WebSocket 二进制帧传输（Binary Frame），比 Base64 编码文本帧体积小 33%

</details>

### 25. SSE 和 WebSocket 的区别？为什么 AI 对话用 SSE，另一个用 WebSocket？

<details>
<summary>查看答案</summary>

| 特性     | SSE                              | WebSocket                    |
| -------- | -------------------------------- | ---------------------------- |
| 方向     | 服务端 → 客户端（单向）          | 双向                         |
| 协议     | HTTP                             | WS/WSS                       |
| 自动重连 | 浏览器原生支持                   | 需手动实现                   |
| 适用场景 | 服务端持续推送（如 AI 流式回复） | 双向交互（如聊天、实时通知） |

- OMS 运维助理用 SSE：因为 AI 对话场景是"用户发一条，AI 回一段"，响应方向是单向的，SSE 够用且实现简单
- 全球机会洞察平台用 WebSocket：因为需要双向通信（用户随时中断对话、发送新消息、服务端推送通知），且需要自定义心跳保活

</details>

### 26. SSE 流式渲染前端怎么实现的？

<details>
<summary>查看答案</summary>

```js
const eventSource = new EventSource("/api/ai/chat?token=xxx");

eventSource.onmessage = (event) => {
  const chunk = JSON.parse(event.data);
  if (chunk.type === "text") {
    // 逐字追加到消息列表
    currentMessage.value += chunk.content;
  } else if (chunk.type === "done") {

  }
```

- 实际项目中用 fetch 实现 POST SSE：

```js
const response = await fetch("/api/ai/chat", {
  method: "POST",
  body: JSON.stringify({ messages }),
  headers: { "Content-Type": "application/json" },
});
const reader = response.body.getReader();
const decoder = new TextDecoder();
while (true) {
  const { done, value } = await reader.read();
  if (done) break;
  const text = decoder.decode(value);
  // 解析 SSE 格式: "data: {...}\n\n"
  parseSSEChunks(text);
}
```

</details>

### 27. WebSocket 断线指数退避重连怎么实现的？

<details>
<summary>查看答案</summary>

```js
class ReconnectingWebSocket {
  constructor(url) {
    this.url = url;
    this.retryCount = 0;
    this.maxRetries = 10;
    this.baseDelay = 1000; // 1s
    this.connect();
  }

  connect() {
    this.ws = new WebSocket(this.url);
    this.ws.onopen = () => {
      this.retryCount = 0; // 连接成功，重置重试计数
      this.flushOfflineQueue(); // 发送离线消息
    };
    this.ws.onclose = () => {
      if (this.retryCount < this.maxRetries) {
        const delay =
          this.baseDelay * Math.pow(2, this.retryCount) + Math.random() * 1000;
        // 1s → 2s → 4s → 8s → 16s → ... 最大约 512s
        setTimeout(() => this.connect(), delay);
        this.retryCount++;
      }
    };
  }

  // 离线消息队列
  send(data) {
    if (this.ws.readyState === WebSocket.OPEN) {
      this.ws.send(data);
    } else {
      this.offlineQueue.push(data); // 断线时缓存消息
    }
  }

  flushOfflineQueue() {
    while (this.offlineQueue.length) {
      this.ws.send(this.offlineQueue.shift());
    }
  }
}
```

- **指数退避**：`delay = baseDelay × 2^retryCount`，避免频繁重连给服务器造成压力
- **随机抖动**：加 `Math.random() * 1000` 防止多个客户端同时重连（惊群效应）
- **Token 自动刷新**：在重连前检查 Token 有效期，过期则先调用刷新接口获取新 Token，再用新 Token 建立连接
- **离线消息队列**：断线期间用户的消息暂存内存队列，重连成功后按顺序补发

</details>

### 28. GraphQL BFF 数据聚合层是怎么搭建的？

<details>
<summary>查看答案</summary>

- **BFF（Backend For Frontend）** 是前端和后端之间的中间层，专门为前端服务
- 在 Alibaba.com 项目中，前端需要同时从多个后端服务获取数据：商品服务、供应商服务、运营配置服务、国家供给服务
- 如果直接调 REST API，一个楼层可能需要 3-4 个接口，PC/Mobile/WAP 三端需要的字段还不一样，导致大量冗余请求
- **GraphQL 方案**：
  - 在 BFF 层定义统一的 GraphQL Schema，聚合所有数据源
  - 前端按需查询：PC 端查 `product { name price image }`，Mobile 端只查 `product { name price }`
  - 一次请求获取所需全部数据，无 Over-fetching / Under-fetching

```graphql
query FloorData($floorId: String!) {
  floor(id: $floorId) {
    products(first: 20) {
      name
      price
      mainImage
      supplier {
        companyName
        country
        yearsOnAlibaba
      }
    }
    operationConfig {
      bannerUrl
      themeColor
    }
  }
}
```

- BFF 层用 Resolver 分别调用各微服务接口，聚合后返回

</details>

### 29. DinamicX 动态化方案是什么？具体怎么工作的？

<details>
<summary>查看答案</summary>

- DinamicX（DX）是阿里内部的动态化渲染框架，核心思想是"模板和数据分离"
- **工作流程：**
  1. 运营在 DX 编辑器中可视化编辑楼层模板（样式、布局、数据绑定规则）
  2. 模板编译后发布到 CDN，生成模板 URL
  3. 服务端下发 `模板URL + 业务数据` 给前端
  4. 前端 DX SDK 下载模板，将业务数据注入模板，运行时渲染出 UI
- **好处：** 运营可以随意调整楼层样式和内容，前端不需要改代码、不需要发版，实现"运营驱动更新"
- **三端统一：** DX 模板引擎同时支持 PC（React 渲染）、Mobile App（Native 渲染）、WAP（H5 渲染），一套模板三端通用

</details>

### 30. OAuth 2.0 单点登录怎么接入的？

<details>
<summary>查看答案</summary>

- 公司 OA 系统作为 Authorization Server，敏捷排期系统作为 Client
- **接入流程（Authorization Code 模式）：**
  1. 用户访问排期系统，未登录 → 重定向到 OA 登录页 `https://oa.company.com/oauth/authorize?client_id=xxx&redirect_uri=xxx&response_type=code`
  2. 用户在 OA 完成登录（可能是域账号 + 密码）
  3. OA 重定向回排期系统，URL 带 `code` 参数
  4. 排期系统后端用 `code` + `client_secret` 向 OA 换取 `access_token`
  5. 用 `access_token` 调用 OA 的用户信息接口，获取 userId、部门、角色
  6. 自动创建/匹配本地用户记录，生成系统 JWT，完成登录
- **组织架构同步：** 登录时从 OA 接口拉取用户所在部门、下属成员，自动同步到本地 User 表，为需求分配提供人员数据

</details>

### 36. 10W+ 行 Vue 单文件重构为轻量 JS 库，具体怎么做的？

<details>
<summary>查看答案</summary>

- 原始情况：参数配置模块是一个巨型 Vue 工程（核心单文件超过 1W+ 行，整体模块累计约 10W+ 行），包含了所有设备类型的参数定义、表单渲染、校验逻辑
- **重构思路：**
  1. 将参数定义（字段名、类型、范围、默认值）抽离为纯 JSON 配置 Schema
  2. 将表单渲染逻辑抽离为通用渲染引擎（接收 Schema → 输出 Vue 组件）
  3. 将校验逻辑抽离为独立校验库（基于 JSON Schema 规范）
  4. 最终产出：一个轻量 JS 库（参数 Schema + 渲染引擎 + 校验器），约 5000 行
- **收益：** 外部团队可以直接引用 JS 库做二次开发，不需要依赖整个 Vue 项目；API 文档用 JSDoc + TypeDoc 自动生成

</details>

### 37. Vue2 → Vue3 迁移怎么做的？怎么保证业务零感知？

<details>
<summary>查看答案</summary>

- **迁移策略：渐进式迁移**
  1. 构建工具替换：Webpack → Vite（用 `@vitejs/plugin-vue2` 先跑通 Vue 2 项目）
  2. 语法兼容层：引入 `@vue/compat`（兼容模式），它让 Vue 3 运行时支持 Vue 2 的语法
  3. 逐步替换：
     - `$on/$off/$once` 事件总线 → mitt 库
     - `filters` → computed 或方法
     - `Vue.prototype` → `app.config.globalProperties`
     - `Vue.set()` → 直接赋值（Vue 3 Proxy 已支持响应式新增属性）
  4. 状态管理：Vuex 3 → Pinia（保持 Options API 写法不变）
  5. UI 组件库：Element UI → Element Plus（组件 API 大部分兼容，少量需手动调整）
- **业务零感知**：保留 Options API 开发模式（`data/methods/computed/watch`），不强制迁移到 Composition API，开发者感知不到底层变化
- 最终关闭 compat 模式，完成完整迁移

</details>

### 38. 首次加载 7s → 2s 的优化手段有哪些？

<details>
<summary>查看答案</summary>

1. **路由懒加载**：首屏只加载当前页面组件，其他页面按需加载
2. **分包策略**：vendor 库独立 chunk，利用浏览器缓存
3. **CDN 外链**：Vue/Element Plus/ECharts 等大库通过 CDN 加载，不走打包（`externals` 配置）
4. **Gzip/Brotli 压缩**：Vite 配置 `vite-plugin-compression`，输出 `.gz` 和 `.br` 文件，Nginx 直接返回
5. **图片优化**：图标用 SVG 雪碧图，大图用 WebP 格式 + 懒加载
6. **预加载**：`<link rel="prefetch">` 预加载可能访问的下一个页面 chunk
7. **DNS 预解析**：`<link rel="dns-prefetch" href="//api.xxx.com">`

</details>

### 39. 异步下载队列（轮询状态机 + 进度追踪）怎么设计的？

<details>
<summary>查看答案</summary>

- **问题**：用户可能同时下载 4-8 路视频，需要控制并发、追踪进度、处理失败重试
- **状态机设计：**

```
IDLE → QUEUED → DOWNLOADING → COMPLETED
                ↓            ↗
              FAILED → RETRYING
```

- **核心实现：**
  - 最大并发数 3（可配置），超出的任务进入等待队列
  - 每个任务有独立的进度条（已下载字节 / 总字节）
  - 轮询服务端下载状态（因为视频下载是服务端异步任务，前端需要轮询）
  - 失败自动重试 3 次，重试间隔指数退避
  - 全局进度 = 所有任务进度的加权平均

</details>
